## Release Boundary

No known implementation or Beads work remains before tagging a private v0.2
candidate. The remaining work is operational validation on Esiban after a tag or
RC artifact exists.

Do not treat this Esiban pass as a substitute for CI. CI already covered
contract goldens, cross-platform tests, archive contents, and Windows Scoop
installation. Esiban is the live Linux dogfood bench: it verifies the tagged
artifact against real local backends, real shell setup, and real qualitative
workflow friction.

## Post-Tag Esiban Validation

After creating the v0.2 RC tag and building release artifacts, run the
project-level Esiban checklist:

```text
../esiban-dogfood-checklist.md
```

Create a release-specific notes file before starting:

```text
../v0.2/esiban-notes-2026-06-13.md
```

Record at least:

- Esiban host metadata: distro, kernel, architecture, and shell.
- Install method: release archive, copied binary, or source build.
- `infer version --json | jq .data`
- `infer capabilities --json | jq .data > /tmp/caps.json` diff result against
  `internal/contract/capabilities.golden.json`.
- Config path used for the Esiban baseline.
- Available backend versions and at least one installed model.
- Pass/fail notes for checklist sections 3 through 14.
- Qualitative notes for checklist section 15.

The v0.1 Esiban pass produced carry-forward fixes that are now implemented in
v0.2: config validation shape, recommendation hygiene, config-subcommand typo
handling, example packaging decision, backend failure semantics, and improved
diagnostic suggestions. If the v0.2 Esiban run finds new issues, open fresh
Beads instead of reopening the closed v0.2 release epic.

## Suggested RC Flow

From a clean checkout on the release commit:

```sh
git status --short
go generate ./internal/contract
scripts/check-contract-goldens.sh
go test ./...
go vet ./...
go build ./...
git tag v0.2.0-rc.1
go run github.com/goreleaser/goreleaser/v2@latest release --clean --skip=publish
scripts/check-release-archives.sh
```

Install the matching Linux artifact on Esiban. If using the archive:

```sh
mkdir -p ~/.local/share/inferctl-0.2.0-rc.1
tar -xzf inferctl_0.2.0-rc.1_linux_amd64.tar.gz -C ~/.local/share/inferctl-0.2.0-rc.1
ln -sfn ~/.local/share/inferctl-0.2.0-rc.1/infer ~/.local/bin/infer
infer version --json
```

If `~/.local/bin` is not on `PATH`, use the PATH guidance in `docs/install.md`
and record the friction in the Esiban notes.

## Minimum Esiban Pass Criteria

Before promoting the RC, these must pass on Esiban:

- Smoke tests: bare `infer`, `--help`, `version`, `version --json`,
  `capabilities --json`, and typo handling.
- Envelope conformance for representative verbs.
- Exit-code checks for invocation, config, lookup, backend timeout, route, and
  validation failures.
- Happy-path `doctor`, `backends`, `models`, `model`, `route`, `config show`,
  `config validate`, and `config explain`.
- v0.2 verbs: `config init`, `config set`, `config patch`, `discover`, and
  `triage`.
- Real Ollama route on Esiban with at least one installed model.
- Source-only examples decision remains acceptable for the artifact being
  tested.

If a MUST item fails, stop and open a blocking Bead with the notes file and exact
command output. If only SHOULD or qualitative items fail, record them and decide
whether to defer to v0.2.1.

## Known Non-Blockers

- Windows behavior is not validated by Esiban. Use GitHub Actions and
  `scripts/smoke-scoop.ps1` for Windows.
- Packaged examples remain deferred; v0.2 ships source-checkout examples only.
- The project remains private-evaluation only until external legal, naming, and
  publication gates clear.
